Android技能学习-触摸事件传递机制

每天学点新知识，抓起老本行。严格意义上来说，学习安卓都快三年了，可是对一些基础知识还是掌握的不好，最近不是很忙，总结一下。

今天学习 Android 中 Activity、ViewGroup、View 三者的触摸事件传递机制

触摸事件的类型

触摸事件对应的是 MotionEvent 类，事件类型主要有如下三种：

ACTION_DOWN

用户手指按下操作，标志着一次触摸事件的开始

ACTION_MOVE

用户手指按下之后在屏幕上移动（非必须，比如用户按了一下屏幕后手指没有移动或立刻离开）

ACTION_UP

用户手指抬起操作，标志着一次触摸事件的结束

触摸事件传递的三个阶段

分发（Dispatch）

事件的分发对应着方法

当前 View 是通过该方法的返回值决定是消费当前触摸事件不再向子视图分发还是不消费当前触摸事件向子视图分发。方法返回 true 表示触摸事件被当前 View 的 onTouchEvent(MotionEvent event) 方法消费，不再继续向子视图分发；返回 false 或 super.dispatchTouchEvent(ev) 表示会继续分发该触摸事件到其子视图。

拦截（Intercept）

事件的拦截对应着方法

方法返回 true 表示触摸事件被当前 View 的 onTouchEvent 方法消费；返回 false 或 super.onInterceptTouchEvent(ev) 表示会继续分发该触摸事件到其子视图。

消费（Consume）

事件的消费  对应着方法

方法返回 true 表示触摸事件被当前 View 的 onTouchEvent(MotionEvent event) 方法消费，触摸事件不会被传递到父视图；返回 false 表示当前视图不处理这个触摸事件，该事件会被传递到父视图的 onTouchEvent 方法进行处理。

触摸事件的传递顺序

  • 触摸事件的传递顺序是由 Activity 到 ViewGroup，再由 ViewGroup 递归地传递给他的子 View；当事件传递到最上层的子 View 后， 又会重新向父 View 传递
  • ViewGroup 是通过 onInterceptTouchEvent 方法来拦截触摸事件，如果该方法返回 true 则事件不会被传递给子 View；如果该方法返回 false 或 super.onInterceptTouchEvent 则事件会继续传递给子 View
  • 如果事件在  向内层传递过程中  由于人工干预（使 onInterceptTouchEvent 方法返回 true）导致事件被拦截并被消费，那么子 View 不会再接收到该事件
  • 如果事件在向外层传递过程中由于人工干预（使 onTouchEvent 返回 true）那么父 View 不会再接收到该事件
  • View 的事件触发顺序是先执行 onTouch 方法再执行 onClick 方法。如果 onTouch 返回 true，那么事件不会再传递，最后也不会调用 onClick 方法
